Analisi riga per riga — main.js

Tamagotchi 3D in WebGL

Questo documento analizza ogni blocco e ogni riga significativa di main.js, spiegando cosa fa il codice, perché è scritto in quel modo e il significato di ogni valore numerico presente. Lo stile segue quello della relazione tecnica principale.

1. Wrapper libreria matematica — m4.js

Le prime righe del file definiscono un livello di "wrapper": funzioni di una sola riga che mappano i nomi interni del progetto alle funzioni della libreria m4.js. Questo permette di usare nomi concisi in tutto il codice (es. mat4Mul) senza cambiare i punti di chiamata se si cambia libreria.

main.js — righe 1–33 — Wrapper m4.js
1
const mat4Mul = (a, b) => m4.multiply(a, b);
Moltiplicazione di due matrici 4×4. Usata per comporre trasformazioni: Model × View × Projection, o per applicare la rotazione della scena a una MVP parziale.
2
const mat4Identity = () => m4.identity();
Restituisce la matrice identità 4×4. Punto di partenza neutro: moltiplicare qualsiasi matrice per l'identità la lascia invariata.
3
const mat4Translate = (tx, ty, tz) => m4.translation(tx, ty, tz);
Crea una matrice di traslazione pura. Usata per posizionare la creatura sull'erba, il pianeta sullo sfondo, sole e luna in orbita.
4
const mat4Scale = (sx, sy, sz) => m4.scaling(sx, sy, sz);
Matrice di scala non uniforme. Tre parametri separati permettono di scalare diversamente per asse, utile ad esempio per schiacciare o stirare un modello. In pratica qui sx=sy=sz (scala uniforme).
5
const mat4RotateY = (rad) => m4.yRotation(rad);
Rotazione attorno all'asse Y verticale. Usata per la rotazione touch/tastiera della scena e per la rotazione lenta del pianeta.
6
const mat4RotateX = (rad) => m4.xRotation(rad);
Rotazione attorno all'asse X orizzontale. Usata per il tilt verticale della scena (drag su/giù) e per ruotare il piano dell'erba da XY a XZ (–π/2).
7
const mat4Perspective = (fovy, aspect, near, far) => m4.perspective(fovy, aspect, near, far);
Costruisce la matrice di proiezione prospettica. I quattro parametri definiscono il frustum visivo: angolo di campo verticale, rapporto larghezza/altezza canvas, piano di clipping vicino e lontano.
8
const mat4TransformPoint = (mat, p) => m4.transformPoint(mat, p);
Trasforma un punto 3D con una matrice 4×4, gestendo la divisione per W. Usata per portare le posizioni di sole e luna dallo spazio di base allo spazio ruotato dalla scena.
9
const vec3Normalize = (v) => m4.normalize(v);
Normalizza un vettore 3D (lunghezza = 1). Usata nell'illuminazione per ottenere direzioni di luce unitarie indipendenti dalla distanza.
10
const vec3Cross = (a, b) => m4.cross(a, b);
Prodotto vettoriale: dati due vettori produce un terzo perpendicolare ad entrambi. Usato internamente da m4.lookAt per costruire la base ortonormale della camera.
11
const vec3Sub = (a, b) => m4.subtractVectors(a, b);
Sottrazione vettoriale componente per componente. Produce il vettore che va da b ad a; usato per calcolare la direzione dalla camera al target in lookAt.
12
function mat4LookAt(eye, center, up) { return m4.inverse(m4.lookAt(eye, center, up)); }
Nota critica: m4.lookAt() restituisce la matrice camera (orientamento della telecamera nello spazio mondo). WebGL però usa la sua inversa come matrice vista. L'inversione è quindi obbligatoria: senza di essa la scena sarebbe capovolta o assente.

2. Riferimenti DOM con jQuery

jQuery seleziona elementi HTML per ID e li memorizza in variabili. Il suffisso [0] è necessario perché le API WebGL e Canvas si aspettano un elemento DOM nativo, non un wrapper jQuery.

main.js — Selezione elementi DOM
—
const canvas = $('#webglcanvas')[0];
Elemento <canvas>: il target del rendering WebGL. Serve l'oggetto nativo per getContext().
—
const feedBtn = $('#feedBtn')[0];
Pulsante "Nutri". Recuperato come riferimento DOM nativo (non strettamente necessario con jQuery ma mantiene coerenza con le API evento touch native sottostanti).
—
const hungerBar = $('#hungerBar')[0]; const moodBar = $('#moodBar')[0];
Elementi <div> delle barre di avanzamento HUD. Aggiornate ogni frame con $(el).css('width', ...).
—
const sizeVal = $('#sizeVal')[0]; const timeVal = $('#timeVal')[0];
Span di testo nell'HUD: mostrano rispettivamente la dimensione corrente della creatura e l'ora simulata del giorno.

3. Contesto WebGL

main.js — Creazione contesto grafico
—
const gl = canvas.getContext("webgl2", { antialias: true }) || canvas.getContext("webgl", { antialias: true }) || canvas.getContext("experimental-webgl");
Strategia a tre livelli di fallback. WebGL2 è preferito (API più ricca). Se non disponibile si usa WebGL1 standard. L'ultimo fallback experimental-webgl copre browser datati. antialias: true attiva il multi-sample antialiasing fornito dalla GPU: rimuove i bordi seghettati senza costo di codice aggiuntivo.

4. Prevenzione zoom del browser

main.js — Event listener globali
—
$(window).on('wheel', (e) => { if (e.originalEvent.ctrlKey) e.preventDefault(); });
Intercetta la rotella del mouse tenendo premuto Ctrl — combinazione che il browser interpreta come "zoom della pagina". e.originalEvent è necessario perché jQuery normalizza l'evento e nasconde le proprietà native dentro originalEvent. preventDefault() blocca lo zoom senza disabilitare lo scroll normale.
—
$(window).on('gesturestart gesturechange gestureend', (e) => e.preventDefault());
Blocca i gesti multitouch nativi di Safari (pizzico per zoomare). Tre eventi coperti in una sola chiamata .on() passando una stringa separata da spazi: caratteristica jQuery che semplifica la registrazione multipla.

5. Funzioni di utilità matematica

main.js — clamp01 e lerp
—
function clamp01(x) { return Math.max(0, Math.min(1, x)); }
Limita x all'intervallo [0, 1]. Usata ovunque si debba garantire che un rapporto (fame, tristezza, fattore giorno) rimanga in range prima di passarlo agli shader come uniform.
—
function lerp(a, b, t) { return a + (b - a) * t; }
Interpolazione lineare: restituisce a quando t=0, b quando t=1, il valore intermedio per t∈(0,1). Usata per transizioni fluide di colore del cielo, intensità luci e dimensione della creatura.

6. Shader della creatura — VS e FS

Gli shader sono stringhe GLSL compilate dalla GPU a run-time. Esistono tre coppie separate, ognuna per un tipo di oggetto diverso.

6.1 Vertex Shader (VS) — creatura

GLSL — vertex shader principale
—
attribute vec3 aPos;
Posizione 3D del vertice, in coordinate modello. Viene dall'OBJ caricato da Blender e inviato alla GPU tramite buffer.
—
attribute vec3 aNor;
Normale del vertice, necessaria per calcolare l'angolo tra la superficie e la luce (illuminazione diffuse Lambertiana).
—
attribute vec3 aCol;
Colore RGB del vertice esportato da Blender (vertex paint). Permette alla creatura di avere colori diversi per ogni parte senza texture aggiuntive.
—
uniform mat4 uMVP;
Matrice Model-View-Projection precomposta sul lato CPU. Trasforma ogni vertice direttamente in clip space (coordinate schermo normalizzate), evitando tre moltiplicazioni separate nella GPU.
—
uniform mat4 uModel;
Solo la matrice modello (senza vista né proiezione). Serve al fragment shader per calcolare la posizione del punto nel world space, necessaria per le luci puntuali (la direzione luce dipende dalla posizione mondiale del punto).
—
vN = mat3(uModel) * aNor;
mat3(uModel) estrae la sottomat 3×3 della matrice modello (solo rotazione e scala, non traslazione). Le normali sono vettori direzionali, non punti: la traslazione non deve influenzarle. Trasformare le normali con la stessa rotazione del modello mantiene coerenza con la geometria deformata.
—
vPos = uModel * vec4(aPos, 1.0);
Posizione del vertice in world space (con W=1.0 perché è un punto, non un vettore). Passata al fragment shader per calcolare la distanza e la direzione verso ciascuna luce puntuale.
—
gl_Position = uMVP * vec4(aPos, 1.0);
Posizione finale in clip space: output obbligatorio di ogni vertex shader. La GPU usa questo valore per rasterizzare il triangolo sullo schermo.

6.2 Fragment Shader (FS) — creatura

GLSL — fragment shader principale
—
precision mediump float;
Precisione floating-point media (16 bit). Scelta per compatibilità con dispositivi mobile che non supportano highp nel fragment shader. Sufficiente per colori e lighting di questa scena.
—
uniform float uSad; // 0..1 uniform float uHunger; // 0..1
Scalari che rappresentano lo stato emotivo della creatura. Valori nell'intervallo [0,1] perché GLSL li usa direttamente in interpolazioni (mix()) e moltiplicazioni di colore. Trasmessi dalla CPU ogni frame.
—
vec3 lightDir = normalize(uLightPos - vPos.xyz); vec3 lightDir2 = normalize(uLightPos2 - vPos.xyz);
Direzione dalla superficie alla sorgente luce. Calcolata per pixel (non per vertice) per un'illuminazione più accurata: su geometrie a bassa densità di vertici, calcolarla per vertice causerebbe sfumature visibilmente errate.
—
float diffuse = max(0.0, dot(n, lightDir));
Modello di illuminazione Lambertiano: la luce ricevuta è proporzionale al coseno dell'angolo tra la normale e la direzione della luce. max(0.0, ...) azzera i valori negativi (lato in ombra): senza questo clamp le facce sul retro riceverebbero luce negativa, sottraendo luminosità ad altri contributi.
—
float fillDiffuse = max(0.0, dot(n, normalize(vec3(0.0, 1.0, 0.35))));
Terza luce "fill" con direzione fissa (dal basso-davanti): 0.0 X, 1.0 Y, 0.35 Z. Simula la luce ambientale riflessa dal terreno. La direzione è fissa in world space perché il suo scopo è riempire le ombre piatte, non seguire la scena.
—
vec3 sadTint = mix(vec3(1.0), vec3(0.75, 0.85, 1.15), uSad); vec3 hungerTint = mix(vec3(1.0), vec3(2.4, 0.7, 0.7), uHunger);
Valori numerici motivati:
sadTint: la tinta triste è blu-freddo: R=0.75 (rosso ridotto), G=0.85 (verde ridotto), B=1.15 (blu aumentato). Il blu supera 1.0 per enfatizzare la sensazione fredda anche su colori già chiari.
hungerTint: R=2.4 spinge il rosso molto in alto (effetto "bruciato di fame"), G e B=0.7 abbassati per aumentare il contrasto con il rosso. Il valore 2.4 è intenzionalmente sopra 1.0: GLSL clampera a 1.0 solo all'output, creando una zona di saturazione rossa visibile su quasi tutti i colori base.
—
float gray = dot(col, vec3(0.299, 0.587, 0.114)); col = mix(col, vec3(gray), uSad * 0.35);
Desaturazione progressiva. I coefficienti 0.299, 0.587, 0.114 sono i pesi di luminanza BT.601 (standard televisivo), che corrispondono alla sensibilità dell'occhio umano ai tre colori primari. Il fattore 0.35 limita la desaturazione massima al 35% per evitare che anche la massima tristezza renda la creatura completamente grigia.

7. Shader texturizzato — VS_TEX e FS_TEX

Coppia separata per erba e pianeta, che campionano una texture UV invece di usare colori vertice.

GLSL — shader texturizzato (erba e pianeta)
—
attribute vec2 aUV;
Coordinate UV (trama): due valori per vertice che indicano quale punto della texture mappa su quel vertice. Due componenti invece di tre perché la texture è 2D.
—
vWorldPos = (uModel * vec4(aPos, 1.0)).xyz;
Posizione mondo passata al fragment shader per calcolare le luci puntuali. Analoga a vPos nel primo shader, ma qui denominata vWorldPos per chiarezza semantica (è già nello spazio mondo, non nel clip space).
—
vec4 texColor = texture2D(uTexture, vUV); if (texColor.a < 0.5) discard;
Test alpha: i pixel con alpha sotto 0.5 vengono scartati senza scrivere nel depth buffer né nel color buffer. Questo permette ai fili d'erba (che hanno trasparenza nei bordi della texture) di apparire ritagliati invece di avere un rettangolo solido visibile. La soglia 0.5 è il punto medio: tutto sotto è "trasparente", tutto sopra è "opaco".

8. Shader emissivo — VS_SUN e FS_SUN

Shader minimale per sole e luna: nessuna illuminazione, solo colore piatto emissivo.

GLSL — shader emissivo (sole e luna)
—
uniform vec3 uSunColor; gl_FragColor = vec4(uSunColor, 1.0);
Il colore è interamente controllato dalla CPU tramite uniform e non dipende da normali né luce. Scelta corretta per sorgenti luminose visibili: una lampada non è illuminata dalla luce, è la luce stessa. L'alpha è fisso a 1.0 (completamente opaco).

9. Compilazione e linking dei programmi shader

main.js — createProgramFromSources via webgl-utils
—
if (window.webglUtils && webglUtils.createProgramFromSources) { program = webglUtils.createProgramFromSources(gl, [VS, FS]); if (!program) throw new Error('...'); } else { /* fallback manuale */ }
Controllo di disponibilità prima di usare la libreria. Il fallback manuale contiene compileShader() e createProgram() che replicano la stessa logica di webgl-utils. In questo modo il codice funziona anche se il file webgl-utils.js non carica (rete lenta, file mancante).
—
let texProgram = webglUtils.createProgramFromSources(gl, [VS_TEX, FS_TEX]); let sunProgram = webglUtils.createProgramFromSources(gl, [VS_SUN, FS_SUN]);
I programmi per texture ed emissivo usano direttamente webgl-utils senza fallback perché se il main program è già stato creato con successo, webglUtils è certamente disponibile. Questo semplifica il codice senza perdere robustezza.
—
gl.deleteShader(vs); gl.deleteShader(fs);
Dopo il linking, i shader compilati non sono più necessari: il programma GPU è autonomo. Liberarli riduce l'utilizzo di memoria sulla GPU.

10. loadTexture — caricamento asincrono texture

main.js — funzione loadTexture
—
return new Promise((resolve, reject) => { ... });
Il caricamento immagini è asincrono per natura (il browser scarica il file in background). La Promise incapsula la callback image.onload in un'interfaccia moderna compatibile con async/await.
—
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_WRAP_S, gl.REPEAT); gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_WRAP_T, gl.REPEAT);
REPEAT fa sì che le UV fuori dall'intervallo [0,1] ripetano la texture invece di estirarne il bordo. Fondamentale per il piano erboso (UV grandi che si ripetono molte volte).
—
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_MIN_FILTER, gl.LINEAR_MIPMAP_LINEAR);
Filtraggio trilineare: interpola tra i due livelli mipmap più vicini E fa interpolazione bilineare all'interno di ciascun livello. Dà la migliore qualità per oggetti lontani, eliminando aliasing e shimmer. Richiede che generateMipmap() sia chiamata.
—
gl.texParameteri(gl.TEXTURE_2D, gl.TEXTURE_MAG_FILTER, gl.LINEAR);
Ingrandimento bilineare: interpola tra i texel vicini quando la texture è ingrandita oltre la sua risoluzione nativa. Evita il pixellato visibile sugli oggetti vicini.
—
gl.generateMipmap(gl.TEXTURE_2D);
Genera automaticamente la catena mipmap: versioni a risoluzione dimezzata (½, ¼, ⅛, …) della texture. Necessarie per il filtro LINEAR_MIPMAP_LINEAR impostato sopra.

11. parseMTL — parser materiali Wavefront

main.js — funzione parseMTL
—
materials[currentMat] = { kd: [0.8, 0.8, 0.8] };
Colore diffuso predefinito grigio chiaro 0.8 su tutti e tre i canali. Scelto perché si vede bene con qualsiasi colore di luce senza essere troppo bianco (che saturerebbe) né troppo scuro. Se un materiale non specifica Kd, il grigio è un fallback neutro visivamente accettabile.
—
if (tag === "Kd" && currentMat) { materials[currentMat].kd = [+parts[1], +parts[2], +parts[3]]; }
Legge solo il canale diffuso (Kd) del materiale. Il file MTL può contenere anche Ka (ambient), Ks (specular), Ns (shininess), ecc., ma il progetto non usa riflessioni speculari né ambient separata, quindi questi campi vengono ignorati per semplicità.

12. parseOBJ — parser geometria Wavefront

Converte il testo OBJ in Float32Array piatti pronti per i buffer GPU. Non usa indici (indexed rendering): ogni vertice della faccia viene espanso in modo indipendente.

main.js — funzione parseOBJ
—
if (parts.length >= 7) { vertColors.push([+parts[4], +parts[5], +parts[6]]); }
Blender può esportare OBJ con colori vertice nella riga v: v x y z r g b. Il controllo length >= 7 detecta questo formato esteso. Se il colore vertice è presente viene usato al posto del colore materiale per quel vertice specifico.
—
outUV.push(t[0], 1.0 - t[1]);
Inversione della coordinata V. Il formato OBJ usa l'origine UV in basso a sinistra; WebGL usa l'origine in alto a sinistra. Invertire V (1.0 - t[1]) corregge questo allineamento: senza questa inversione le texture apparirebbero capovolte verticalmente.
—
// triangolazione a ventaglio const verts = parts.slice(1).map(v => v.split("/")); for (let i = 1; i + 1 < verts.length; i++) { const tri = [verts[0], verts[i], verts[i+1]]; }
Triangolazione a ventaglio (fan triangulation): il primo vertice della faccia diventa il centro, e ogni coppia successiva di vertici forma un triangolo con esso. Questo funziona correttamente per facce convesse (quad Blender), che è il caso tipico. WebGL non supporta poligoni, solo triangoli.
—
function parseIndex(str, len) { const idx = parseInt(str, 10); return idx >= 0 ? (idx - 1) : (len + idx); }
Il formato OBJ usa indici 1-based (il primo elemento è 1, non 0): quindi idx - 1 converte in 0-based per gli array JavaScript. Gli indici negativi sono relativi alla fine della lista corrente (len + idx): usati da alcuni esportatori per riferirsi ai vertici appena definiti.
—
return { pos: new Float32Array(outPos), ... vertCount: outPos.length / 3 };
Float32Array invece di array normale: i buffer WebGL richiedono dati binari tipizzati. La GPU li interpreta come sequenza di float a 32 bit. vertCount è il numero di vertici totali (non il numero di float), calcolato dividendo la lunghezza dell'array posizioni per 3 (tre float per vertice: x, y, z).

13. createArrayBuffer e bindAttrib

main.js — gestione buffer GPU
—
return webglUtils.createBufferFromTypedArray(gl, data, gl.ARRAY_BUFFER, gl.STATIC_DRAW);
gl.ARRAY_BUFFER: tipo buffer per attributi vertice (contrapposto a ELEMENT_ARRAY_BUFFER per gli indici). gl.STATIC_DRAW: suggerimento alla GPU che i dati saranno scritti una volta e letti molte volte — permette ottimizzazioni di allocazione nella VRAM.
—
function bindAttrib(name, size, buffer) { const loc = gl.getAttribLocation(program, name); gl.vertexAttribPointer(loc, size, gl.FLOAT, false, 0, 0); }
size è il numero di componenti per vertice (3 per posizione/normale/colore, 2 per UV). gl.FLOAT: tipo dati. false: non normalizzare (i dati sono già in range corretto). Stride 0: gli attributi sono in buffer separati, non interleaved (stride=0 dice alla GPU di dedurre il passo dal size). Offset 0: i dati iniziano dall'inizio del buffer.

15. Oggetto di stato (state)

Unico punto di verità per tutto lo stato della simulazione. Raccogliere lo stato in un oggetto evita variabili globali sparse e facilita il debug (basta ispezionare state dalla console).

ProprietàValore inizialeSignificato e motivazione
t, lastT0Tempo corrente e precedente in secondi. La differenza è dt (delta time del frame), necessario per aggiornamenti indipendenti dal frame rate.
hunger0.0Fame continua [0..1] per gli shader. Inizia a 0 (sazio) e sale nel tempo.
hungerInt0Contatore intero [0..20]: versione discreta della fame per la logica di gioco. Più facile da ragionare rispetto a valori float continui.
nextHungerAtMsDate.now() + 5000Timestamp wall-clock per il prossimo incremento fame. Usa Date.now() (ms reali) invece di state.t (tempo animazione) per essere resiliente ai tab in background che rallentano requestAnimationFrame.
moodLevel0Umore discreto [0=Felice .. 20=Triste]. Stesso schema a 20 passi della fame per coerenza visiva nelle barre HUD.

targetSize1.0Dimensione target verso cui size si avvicina gradualmente. Separare target e valore corrente permette transizioni smooth senza codice di animazione esplicito.

17. Controlli touch

main.js — addEventListener touch (versione nativa)
—
const s = 0.006; state.rotY += dx * s; state.rotX += dy * s;
Sensibilità touch: 0.006 radianti per pixel. Con questo valore, trascinare su tutta la larghezza di uno schermo da ~400px produce una rotazione di ~2.4 rad (≈137°), che è una svolta sostanziale ma non eccessiva. Valori più alti renderebbero la creatura difficile da controllare con precisione.
—
state.rotX = Math.max(-0.6, Math.min(0.9, state.rotX));
Clamp della rotazione verticale: -0.6 rad (≈-34°, guarda da sotto) e 0.9 rad (≈51°, guarda dall'alto). Senza questo limite, l'utente potrebbe ruotare completamente sopra o sotto la scena, perdendo l'orientamento. Il limite superiore (0.9) è più permissivo per permettere una vista dall'alto più ampia.
—
{ passive: false }
Opzione obbligatoria per poter chiamare e.preventDefault() dentro l'handler touch. I browser moderni registrano i listener touch come "passivi" di default (per ottimizzare lo scroll): questo flag dice esplicitamente che l'handler può bloccare il comportamento default.

18. Controlli tastiera

main.js — keydown con jQuery
—
const KEYBOARD_ROTATION_SPEED = 0.05;
Radianti per pressione tasto: 0.05 rad ≈ 2.9° per pressione. Tenendo premuta la freccia a ~60 pressioni/secondo si ottiene ~3 rad/s. Valore simile alla rotazione auto (0.9 rad/s) ma più rapido per consentire controllo manuale veloce.
—
$(document).on('keydown', (e) => { ... e.preventDefault(); });
e.preventDefault() blocca lo scroll della pagina causato dai tasti freccia. Senza di esso, premere "Giù" scorrebba la pagina invece di ruotare la scena.

19. Pulsanti Nutri e Conforta

main.js — gestori pulsanti
—
$('#feedBtn').on('click', () => { state.hungerInt = state.hungerInt - 1; state.hunger = clamp01(state.hungerInt / 20); });
Ogni clic riduce la fame di 1 unità (un passo discreto). La scelta di un singolo passo invece di un reset completo bilancia il gameplay: serve tanta interazione quanta ne accumula la fame. clamp01 impedisce che hunger diventi negativo se si clicca quando la creatura è già sazia.
—
$('#cheerBtn').on('click', () => { state.moodLevel = Math.max(0, state.moodLevel - 1); state.nextMoodAtMs = Date.now() + 5000; });
Confortare resetta anche il timer del prossimo peggioramento umore (+5000ms). Questo premia l'interazione: non solo migliora l'umore di uno step, ma "resetta l'orologio" del degrado successivo, dando alla creatura un momento di respiro.

20. Pannello dat.GUI

main.js — dat.GUI setup
—
if (window.dat && dat.GUI) { const gui = new dat.GUI(); }
Controllo di disponibilità: se dat.GUI non è caricato il codice continua a funzionare senza il pannello. La GUI è uno strumento di sviluppo opzionale, non una dipendenza critica.

21. createSphereMesh e createTexturedSphereMesh

Due generatori procedurali di sfere. Il primo è per sole e luna (solo posizioni), il secondo per il pianeta (posizioni + normali + UV).

main.js — generatori sfere procedurali
—
for (let lat = 0; lat < latBands; lat++) { const t0 = (lat / latBands) * Math.PI; const t1 = ((lat + 1) / latBands) * Math.PI;
Suddivisione della sfera in bande latitudinali (da polo a polo). Ogni banda va da un angolo theta0 a theta1, entrambi nell'intervallo [0, π]. Il valore π è il range completo dell'angolo polare.
—
function sph(r, theta, phi) { const sinT = Math.sin(theta); return [ r * sinT * Math.cos(phi), r * Math.cos(theta), r * sinT * Math.sin(phi) ]; }
Conversione da coordinate sferiche a cartesiane: X = r·sinθ·cosφ, Y = r·cosθ, Z = r·sinθ·sinφ. Questa formula è la definizione standard delle coordinate sferiche. Y è l'asse verticale (convenzione WebGL/OpenGL con Y verso l'alto).
—
const sunMesh = createSphereMesh(1.0, 20, 20); const moonMesh = createSphereMesh(1.0, 16, 16);
Sole: 20×20 bande = 2400 triangoli — abbastanza risoluto da sembrare rotondo. Luna: 16×16 = 1536 triangoli — leggermente meno dettagliata perché più piccola a schermo (dimensione scala 0.28 vs 0.35 del sole).
—
const planetMesh = createTexturedSphereMesh(1.0, 24, 24);
Il pianeta usa 24×24 bande (2304 quad = 4608 triangoli): più dettagliato perché la texture richiede sufficiente densità di UV per campionare correttamente. Con meno suddivisioni la texture apparirebbe distorta vicino ai poli.
—
const invLen = 1.0 / (Math.hypot(p[0], p[1], p[2]) || 1.0);
Calcola l'inversa della lunghezza del vettore (evitando la divisione per zero con || 1.0). Per una sfera di raggio 1.0 il vettore è già normalizzato, ma questo calcolo è corretto per qualsiasi raggio.

22. Funzione init() — caricamento asincrono asset

main.js — IIFE asincrona di inizializzazione
—
(async function init() { try { ... } catch(err) { ... } })();
IIFE (Immediately Invoked Function Expression) async: si chiama da sola al caricamento della pagina. Il blocco try/catch cattura qualsiasi errore (rete, parse, shader) e mostra un messaggio leggibile invece di un errore silenzioso nella console.
—
const mtlText = await loadText("assets/creature.mtl"); const materials = parseMTL(mtlText); const objText = await loadText("assets/creature.obj"); mesh = parseOBJ(objText, materials);
Il file MTL viene caricato prima dell'OBJ. Necessario perché il parser OBJ ha bisogno dei materiali durante il parsing delle facce (usemtl). Se si caricassero in ordine inverso, tutti i colori sarebbero il grigio di default.
—
for (const matName in grassMaterials) { grassMaterials[matName].kd = [0.2, 0.6, 0.15]; }
Sovrascrive il colore di tutti i materiali dell'erba con verde scuro [0.2, 0.6, 0.15]. La texture dell'erba già porta la variazione visiva; il colore materiale serve come base moltiplicativa per l'illuminazione. Il verde specifico è calibrato per darsi con i colori della luce diurna.
—
gl.enable(gl.DEPTH_TEST); gl.disable(gl.CULL_FACE);
DEPTH_TEST: attiva il depth buffer — ogni pixel scritto viene confrontato con la profondità precedente; vince il più vicino. Senza questo, gli oggetti si sovrapporrebbero in ordine di draw call invece che per distanza reale. CULL_FACE disabilitato: mostra entrambi i lati dei poligoni — necessario per i fili d'erba che sono quad a singolo layer (senza face culling mostrerebbero solo la faccia verso la camera).

23. computeBounds — calcolo bounding box

main.js — computeBounds
—
let minX = Infinity, maxX = -Infinity; ...
Inizializzare con Infinity e -Infinity garantisce che il primo valore incontrato (qualunque esso sia) diventi immediatamente il nuovo minimo o massimo. È il pattern standard per trovare min/max in un passaggio unico senza assumere nulla sull'intervallo dei dati.
—
const radius = Math.max(dx, dy, dz) * 0.5;
Raggio della bounding sphere: metà della dimensione massima tra le tre direzioni. Usato per posizionare il fondo della creatura esattamente sull'erba: bounds.minY * scale è l'offset dal centro del modello al suo piede inferiore.

24. Ciclo di animazione — loop()

main.js — loop principale
—
const t = tsMs * 0.001;
Conversione millisecondi → secondi. 0.001 = 1/1000. I timestamp di requestAnimationFrame sono in millisecondi; tutto il codice fisico usa secondi perché le velocità (rad/s, unità/s) sono più intuibili in quella unità.
—
const dt = Math.min(0.05, Math.max(0.0001, t - state.lastT));
Delta time con doppio clamp. Massimo 0.05 s (= 20 fps minimo effettivo): se il tab è in background e i frame si accumulano, senza questo clamp un singolo dt di 5 secondi causerebbe un salto enorme nello stato (fame che avanza di cinque volte, creatura che scala istantaneamente). Minimo 0.0001 s: evita divisioni per zero in eventuali calcoli basati su 1/dt.
—
update(dt); render(); requestAnimationFrame(loop);
Separazione update/render: la logica di gioco (update) è indipendente dalla logica grafica (render). La coda al prossimo frame viene sempre richesta in fondo, anche dopo un eventuale errore nel render, per mantenere il loop attivo.
—
if (webglUtils.resizeCanvasToDisplaySize(canvas, window.devicePixelRatio)) { gl.viewport(0, 0, canvas.width, canvas.height); }
devicePixelRatio è 2 su schermi Retina/HiDPI: il canvas fisico è il doppio dei CSS pixel. resizeCanvasToDisplaySize adatta le dimensioni del buffer di rendering al display reale, evitando rendering sfocato su display ad alta densità. gl.viewport() va aggiornato di conseguenza per mappare correttamente clip space → pixel fisici.

25. Funzione update() — logica di gioco

main.js — update(dt)
—
const steps = Math.floor((nowMs - state.nextHungerAtMs) / 5000) + 1; state.hungerInt = Math.min(HUNGER_MAX, state.hungerInt + steps);
Sistema a step: non incrementa per frazione di secondi, ma conta quanti step da 5000ms sono passati. Questo garantisce che anche se il browser perde molti frame (tab in background) la fame avanzi esattamente il numero corretto di passi. Il +1 aggiunge lo step corrente al conto degli step passati.
—
const MIN_SIZE = 0.15; const MAX_SIZE = 0.35; state.targetSize = lerp(MIN_SIZE, MAX_SIZE, 1 - sizeHunger);
MIN_SIZE = 0.15: la creatura al massimo della fame è molto piccola (15% della scala modello). MAX_SIZE = 0.35: a stomaco pieno è al 35% della scala. Questi valori tengono la creatura visibile e riconoscibile in entrambi gli stati senza essere troppo grande da occupare tutto lo schermo o troppo piccola da non vederla. 1 - sizeHunger inverte la mappatura: fame=1 → size=MIN, fame=0 → size=MAX.
—
const k = 1 - Math.exp(-RATES.sizeSmoothPerSec * dt); state.size = lerp(state.size, state.targetSize, k);
Smoothing esponenziale a frame-rate indipendente. La formula k = 1 - e^(-rate·dt) è equivalente matematicamente a un filtro low-pass: garantisce la stessa velocità di convergenza indipendentemente dal frame rate. Con rate=3.0: il 95% del cambiamento avviene in ~1 secondo a qualsiasi fps.
—
const hungerPercent = Math.min(100, (state.hungerInt / HUNGER_MAX) * 100); $(hungerBar).css({ width: hungerPercent + '%', backgroundPosition: hungerPercent + '% 0' });
La larghezza della barra è la fame in percentuale. backgroundPosition sposta anche la posizione del gradiente di sfondo della barra in modo proporzionale, creando un effetto visivo dove il colore del gradiente "segue" il riempimento.
—
const clockHours = (dayPhase * 24.0 + CLOCK_START_HOUR) % 24.0;
Converte la fase del ciclo [0,1] in ore [0,24], poi aggiunge l'offset di partenza (8.0 = 8:00) con modulo 24 per gestire il wrap-around. La moltiplicazione per 24.0 scala un giorno in-game a 24 ore visualizzate.
—
if (clockHours >= 5.0 && clockHours < 8.0) periodLabel = 'Dawn'; else if (clockHours >= 8.0 && clockHours < 17.0) periodLabel = 'Day'; else if (clockHours >= 17.0 && clockHours < 20.0) periodLabel = 'Dusk';
Fasce orarie: alba 5-8, giorno 8-17, crepuscolo 17-20, notte per il resto. Corrispondono approssimativamente a orari reali del sole alle medie latitudini. L'orario 8 e 17 (9 ore di giorno) è leggermente più corto del giorno estivo reale ma più intuitivo.
—
state.rotY += dt * 0.9;
Velocità auto-rotazione: 0.9 rad/s ≈ 51°/s → un giro completo ogni ~7 secondi. Abbastanza lento da apprezzare tutti i lati della creatura, abbastanza rapido da non sembrare fermo.

26. Funzione render() — pipeline di disegno

26.1 Camera e proiezione

main.js — setup camera
—
const projectionMatrix = mat4Perspective(Math.PI / 4, aspectRatio, 0.1, 100.0);
Parametri frustum motivati:
Math.PI / 4 = 45° fov verticale: angolo di campo standard per applicazioni 3D. Più stretto di 60° (meno distorsione prospettica), più largo di 30° (campo visivo sufficientemente ampio).
0.1 near plane: oggetti più vicini di 0.1 unità vengono clippati. Troppo piccolo causa problemi di precisione depth (z-fighting); 0.1 è il compromesso comune.
100.0 far plane: tutto entro 100 unità è visibile. Il pianeta più lontano è a circa 30 unità (√(16²+8.5²+26²)≈32), quindi 100 ha margine sufficiente.
—
const camEye = [state.camX, state.camY, state.camZ]; // default [0,0,13.3] const camLook = [0, 0, 0]; const camUp = [0, 1, 0];
La camera guarda sempre verso l'origine [0,0,0] (dove si trova la creatura). camUp = [0,1,0] definisce la direzione "su" della camera come l'asse Y verticale — standard OpenGL/WebGL.

26.2 Orbita sole e luna

main.js — orbita celeste
—
const orbitAngle = dayPhase * Math.PI * 2.0 + Math.PI * 0.5; const moonAngle = orbitAngle + Math.PI;
dayPhase * 2π mappa [0,1] su [0,2π] per un ciclo completo. + π/2 sfasa l'orbita di 90° così che l'alba (fase=0) inizia con il sole appena sull'orizzonte est invece che nel punto più alto. moonAngle = orbitAngle + π mette la luna esattamente all'opposto del sole: quando il sole tramonta, la luna sorge.
—
const sunBasePos = [ Math.cos(orbitAngle) * ORBIT_RADIUS_X, ORBIT_CENTER_Y + Math.sin(orbitAngle) * ORBIT_RADIUS_Y, 2.4 ];
Il sole orbita in un piano leggermente inclinato verso la camera (Z=2.4). Questo evita che il sole passasse esattamente dietro la scena ad ogni ciclo, rendendolo sempre parzialmente visibile. La Z positiva porta il sole in primo piano durante il giorno.
—
const dayFactor = clamp01((sunBasePos[1] + ORBIT_RADIUS_Y * 0.15) / (ORBIT_RADIUS_Y * 1.15));
Fattore giorno [0,1] basato sull'altezza Y del sole. Il termine ORBIT_RADIUS_Y * 0.15 sposta la soglia dell'alba/tramonto leggermente sotto lo zero: il cielo comincia a illuminarsi prima che il sole emerga visivamente dall'orizzonte, simulando la luce diffusa atmosferica. Il denominatore ORBIT_RADIUS_Y * 1.15 normalizza il range per ottenere un fattore che raggiunge effettivamente 1.0 al massimo dell'orbita.

26.3 Colori dinamici del ciclo giorno/notte

main.js — colori cielo e luci
—
const daySky = [0.44, 0.62, 0.86]; const nightSky = [0.04, 0.06, 0.12];
daySky: azzurro chiaro (R basso, G medio, B alto — cielo sereno). nightSky: quasi nero con leggera tinta blu [0.04, 0.06, 0.12] — il blu notte è leggermente sopra il nero puro per dare profondità senza essere completamente piatto.
—
const sunLightColor = [ 1.05 * dayFactor, 0.95 * dayFactor, 0.78 * dayFactor ];
Colore luce solare: R=1.05, G=0.95, B=0.78 moltiplicati per dayFactor. I valori R e G leggermente sbilanciati danno alla luce solare una tinta calda giallo-arancio, mentre B=0.78 riduce il blu (caratteristica della luce diurna reale). Tutti moltiplicati per dayFactor si azzerano di notte.
—
const moonLightColor = [ 0.24 * nightFactor, 0.34 * nightFactor, 0.54 * nightFactor ];
Luce lunare: tinta fredda blu-viola (R<G<B). 0.24, 0.34, 0.54 sono tutti under 0.6 (luce debole rispetto al sole che arriva a 1.05). La luna di notte illumina debolmente con luce fredda — sensazione esteticamente appropriata e coerente con la percezione visiva reale.
—
const ambientColor = [ lerp(0.06, 0.22, dayFactor), lerp(0.07, 0.24, dayFactor), lerp(0.10, 0.28, dayFactor) ];
Luce ambientale: interpolata tra notte [0.06, 0.07, 0.10] e giorno [0.22, 0.24, 0.28]. Di notte l'ambiente è molto scuro ma non zero (le stelle illuminano leggermente). Il giorno è chiaramente illuminato. I valori di giorno hanno più blu che rosso per il cielo blu diffuso.
—
const fillLightColor = [ lerp(0.20, 0.06, dayFactor), lerp(0.24, 0.07, dayFactor), lerp(0.32, 0.10, dayFactor) ];
Fill light invertita rispetto all'ambiente: di notte è più intensa [0.20, 0.24, 0.32] con tinta blu-viola (simula la luce del cielo notturno vista dal basso). Di giorno si abbassa [0.06, 0.07, 0.10] perché la luce solare diretta già illumina tutto. L'inversione evita che la fill light e la sun light si sommino in modo eccessivo di giorno.

26.4 Creatura: scala, posizionamento e animazione idle

main.js — matrice modello creatura
—
const planeY = -1.5; const grassSurfaceY = planeY + 0.8;
planeY = -1.5: il centro del piano dell'erba è a -1.5 unità sotto l'origine. + 0.8: stima dell'altezza della superficie erbosa (altezza modello erba ~10 unità × scala 0.04 × rotazione = ~0.8 unità verticali). La creatura si appoggia a questa altezza stimata.
—
const creatureBottomOffset = bounds.minY * meshScale;
Il minY dei bounds è la coordinata Y del punto più basso del modello in coordinate modello. Moltiplicato per la scala attuale dà la distanza dal centro al piede inferiore in coordinate mondo. Sottrarlo dalla posizione Y mette i piedi esattamente sull'erba anziché il centro del modello.
—
const bobAmount = 0.02; // unità const bobSpeed = 2.0; // rad/s const breatheAmount = 0.015; // percentuale scala const breatheSpeed = 1.5; // rad/s
Parametri animazione idle motivati:
bobAmount = 0.02: oscillazione di 2 cm in scala mondo — percettibile ma non drammatica. Troppo alta (>0.1) sembrerebbe un salto, troppo bassa (<0.005) sarebbe invisibile.
bobSpeed = 2.0 rad/s: ~0.3 Hz → un'oscillazione ogni ~3 secondi. Ritmo simile a un respiro tranquillo o galleggiamento.
breatheAmount = 0.015: scala ±1.5% — appena visibile, simula l'espansione del corpo durante la respirazione.
breatheSpeed = 1.5 rad/s: leggermente più lento del bob per un ritmo asincrono che evita la sensazione meccanica di movimento perfettamente sincronizzato.
—
const bobOffset = Math.sin(idleTime * bobSpeed) * bobAmount; const breatheScale = 1.0 + Math.sin(idleTime * breatheSpeed) * breatheAmount;
Il seno oscilla tra -1 e +1 con periodo 2π/speed. Moltiplicato per l'ampiezza dà l'offset effettivo. Il 1.0 + davanti al seno del respiro trasforma l'oscillazione da [-1,+1] a [0.985, 1.015], cioè una scala che oscilla intorno a 1.0.

26.5 Erba — piano texturizzato

main.js — trasformazione piano erba
—
const grassScale = 0.04; const rotX90 = mat4RotateX(-Math.PI / 2); const translateM = mat4Translate(0, planeY, 0); const grassModel = mat4Mul(translateM, mat4Mul(rotX90, scaleM));
Trasformazioni in ordine (applicato da destra a sinistra):
1. scaleM (0.04): il modello OBJ dell'erba ha coordinate nell'ordine delle centinaia di unità; scalarlo a 0.04 lo porta a ~5-6 unità nel piano XY.
2. rotX90 (-π/2): il piano OBJ è nel piano XY (orizzontale rispetto a Blender con Z in alto). Ruotare di -90° attorno a X fa giacere il piano nel piano XZ (orizzontale rispetto a WebGL con Y in alto).
3. translateM (0, -1.5, 0): sposta il piano al livello del terreno.

26.6 Pianeta distante

main.js — posizione e rotazione pianeta
—
const planetScale = 2.6; const planetSpin = mat4RotateY(state.t * 0.12); const planetModel = mat4Mul( mat4Translate(-16.0, 8.5, -26.0), mat4Mul(planetSpin, mat4Scale(2.6, 2.6, 2.6)) );
Valori numerici motivati:
Posizione [-16, 8.5, -26]: il pianeta è lontano a sinistra e in alto (X=-16), relativamente alto (Y=8.5) per essere visibile sopra l'orizzonte, profondo in Z=-26 per dare sensazione di distanza. Con near=0.1 e far=100.0, questa distanza è ben dentro il frustum.
Scala 2.6: la sfera di raggio 1.0 × 2.6 = raggio 2.6 unità. A distanza ~32 unità appare come un piccolo disco, convincente come pianeta di sfondo.
Velocità rotazione 0.12 rad/s: un giro ogni ~52 secondi, quasi uguale alla durata del giorno (60s) — deliberatamente lento per sembrare astronomicamente plausibile.

26.7 Sole e luna — sfera emissiva

main.js — rendering sole e luna
—
const sunScale = 0.35; const moonScale = 0.28;
Sole 0.35, luna 0.28: la luna è ~20% più piccola del sole — approssimazione della percezione visiva reale (nel cielo reale il sole e la luna appaiono quasi identici per dimensioni angolari, ma la luna leggermente più piccola in questa scena stilizzata è una scelta artistica).
—
gl.uniform3f(sunUColor, lerp(0.55, 1.0, dayFactor), lerp(0.55, 0.92, dayFactor), lerp(0.65, 0.52, dayFactor) );
Colore visivo del sole interpolato tra notte [0.55, 0.55, 0.65] (bianco-blu, come il sole all'orizzonte) e giorno pieno [1.0, 0.92, 0.52] (giallo-bianco brillante). Di notte il sole appare biancastro perché è sotto l'orizzonte ma lo shader non lo sconde — il modello di visibilità è lasciato al ciclo orbitale.
—
gl.uniform3f(sunUColor, // luna lerp(0.65, 0.25, dayFactor), lerp(0.70, 0.32, dayFactor), lerp(0.82, 0.45, dayFactor) );
Colore luna: di notte [0.65, 0.70, 0.82] — bianco freddo con tinta blu (luce lunare riflessa). Di giorno [0.25, 0.32, 0.45] — più scuro e quasi invisibile nel cielo diurno. Il blue domina in entrambi i casi per la natura fredda della luce lunare.
—
const err = gl.getError(); if (err !== 0) console.warn('GL error after draw:', err);
Controllo errori WebGL alla fine di ogni frame. gl.getError() restituisce 0 (= gl.NO_ERROR) se tutto va bene. Chiamarla ogni frame ha un costo minimo di prestazioni ma permette di identificare rapidamente errori GPU durante lo sviluppo (buffer non collegati, uniform mancanti, ecc.).
